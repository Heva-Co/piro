Question
DECISION: Is the check-dimension ↔ AlertConfig binding proposal for Piro sound end-to-end, or does it have structural flaws that should change its shape before it is built and made hard to revert? Proposal: (1) DimensionSpec(Name, ThresholdDirection Direction, Unit) reusable, used in both the manifest and when measuring so the dimension name cannot diverge; (2) AlertConfig.AlertFor enum -> AlertConfig.Dimension string; (3) AlertEvaluationService switch -> generic loop comparing by ThresholdDirection (HigherIsWorse => value>=threshold; LowerIsWorse => value<=threshold); (4) delete the AlertFor enum. CheckDataPoint (high-volume time-series, one MetricValue+Status+LatencyMs) would migrate to N dimensions. Options: A adopt as-is; B keep Status out of the dimension system; C enrich direction to band/range/equality now; D defer the hot-table migration. Prior council eac976ec (same contract) recommended the fixed-fields model plus a named-metrics hook toward per-dimension evaluation.
Pre-council bias scan (signals, not verdicts) Sunk cost / IKEA — "already approved and coded" framed as reasons the shape is settled. Anchoring — the four-part proposal presented fully-formed, inviting critique-in-place over alternatives. Confirmation — asks to validate soundness "end-to-end". Status-quo/momentum — "hard to revert" and "user mandated full multi-dimension" treated as fixed. Narrative — the clean "hook -> concrete realization" story makes the path feel inevitable.
Council Verdict

Recommendation

Adopt the core of the proposal, but reshape it on four points and split the irreversible parts out. The DimensionSpec dedup (one declaration feeding both manifest and measurement so a name cannot be mistyped) and the string-dimension binding with a generic evaluator are genuinely sound and are the real win. What is unsafe is bundling them with two hard-to-revert moves and asserting a comparison model that is one notch too narrow.

The four reshapes

  • Keep Status out of the direction model. Status is UP/DOWN, evaluated today by enum equality (dp.Status == targetStatus), not by >=/<=. Forcing it into HigherIsWorse/LowerIsWorse is, in the Executor's words, "a lie encoded into data." Status is the result envelope, not a numeric dimension. Model it as a first-class equality predicate; let the dimension system own only the numeric lane. This is the council's single strongest point of agreement.
  • Model comparison as a small typed set, not direction alone. Ship Threshold(direction) + Equality now (Equality carries Status cleanly), leave room for Band/Range later. Direction stays the common case; it stops pretending to be the only case. This absorbs the SSL "expired vs expiring" and equality cases without a second rewrite.
  • Retire AlertFor with string-compatibility; do not hard-delete it. AlertFor is not an internal enum — it is a user-facing YAML config-as-code field (parsed in YamlImportService), an OpenAPI/DTO type, and ~6 admin-UI files plus web api-types. A hard delete silently breaks users' committed YAML. Replace the enum with the Dimension string but keep parsing the old values so existing config keeps importing.
  • Split the CheckDataPoint migration into its own PR and defer it. The hot composite-PK time-series table stores a single MetricValue slot today, and every current check alerts on exactly one numeric dimension, so nothing forces the N-dimension fan-out yet. Do the DimensionSpec + string-binding + generic-evaluator refactor as one reversible PR, keeping one AlertConfig = one dimension. Migrate storage only when a check genuinely needs to alert on 2+ numeric dimensions at once.

Why this beats adopting as-is

Option A couples a cheap reversible refactor with the two expensive irreversible moves (enum delete + hot-table migration) on the change the user calls "the last one." It also encodes the Status lie, which returns as a hidden if (isStatus) branch inside the "generic" loop — less honest than the switch it replaced. The reshaped path gets every win the proposal wanted (name-cannot-diverge, enum killed from code, generic evaluator) while keeping the irreversible pieces behind their own decision.

What survived the debate

The Prosecutor's sharpest attack was that deleting AlertFor dissolves the "streak of what?" identity behind FailureThreshold's consecutive-breach counter, silently mis-counting breaches. Ground truth disproved the headline: the streak is computed per-AlertConfig and keyed on config.Id (via config.IsAlerting), never on AlertFor — the rename is identity-neutral. But the Prosecutor's probe surfaced the one real residual risk, which reinforces reshape #4: historical data points carry only a single MetricValue, so two configs alerting on two numeric dimensions of the same check cannot both reconstruct their consecutive-failure window from history. That is a storage-shape constraint, and it is exactly why the migration must be its own, later, deliberately-sequenced PR.

First thing to do

Ship PR 1 (reversible): DimensionSpec catalog + AlertConfig.Dimension string with backward-compatible AlertFor parsing + a small typed comparison set (Threshold+Equality) + the generic evaluator, with Status on the equality path and one-dimension-per-config. Leave CheckDataPoint untouched. Hold the storage migration as PR 2, gated on a real multi-numeric-dimension consumer.

Dissent Ledger — minority signal the consensus softened
  • The "two-lane" framing (Expansionist) is stronger than a Status carve-out. Consensus treats "keep Status out" as an exception bolted onto option A. The minority view: make it the organizing principle — a numeric dimension lane and a categorical/status lane by construction — so no special-casing is needed later. Worth adopting as the mental model even while implementing the typed-comparison-set version.
  • The parsed-string AlertValue is an unaddressed "units violation" (Outsider). Even after the reshape, the threshold is still a stringly-typed value parsed at evaluation, so Direction and the value's shape live in two places that can silently disagree. A typed AlertValue (StatusMatch | NumericThreshold) was proposed and not folded into the recommendation; revisit it when the comparison set grows.
  • Alert-history / incident continuity across the AlertFor->Dimension identity change went un-analyzed. Two reviewers flagged that renaming the discriminator could orphan or double-fire active incidents. The streak keys on config.Id so open alerts are safe, but confirm no incident/dedup key is derived from the AlertFor string before shipping PR 1.
Debate Round (triggered — consensus avg 4.0/5)
Prosecutor The lethal flaw: the consensus treats evaluation as stateless f(value, threshold, direction), but Piro counts consecutive breaches (FailureThreshold / N-of-M). That counter needs a stable identity to accumulate against, and today AlertFor IS that join key — "streak of what?" Retire it to a free string across N dimensions and the streak's subject dissolves; the window logic gets smuggled back around the loop and silently mis-counts, so pages fire late or never. Confirming test: grep the evaluator for a consecutive-failure counter keyed on AlertFor.
Ground truth (AlertEvaluationService.cs) The streak is computed per-AlertConfig: EvaluateConfigAsync(config) runs once per config, CountConsecutive(recentPoints, conditionMet), keyed on config.Id via config.IsAlerting — never on AlertFor. The Prosecutor's confirming grep FAILS for them.
Defender CONCEDE: historical CheckDataPoint rows carry one MetricValue, so two configs on two numeric dimensions of the same check cannot both reconstruct their consecutive-window history — this vindicates deferring the storage migration. REBUT: the streak keys on config.Id, not AlertFor, so the rename is identity-neutral; the headline attack is disconfirmed. REMAINING RISK: the multi-numeric-dimension history-read gap — monitor by asserting one AlertConfig = one dimension and gating any second numeric-dimension config on the migration landing first; pin Dimension strings to a DimensionSpec catalog so they can't drift per-probe.
Advisor Responses
Red Team high
Verified 3 facts: Status is string-equality (Enum.TryParse + ==), not >=/<=; MetricValue is ALREADY a single generic slot shared by CertExpiry + FailedNameServers, so no N-column migration exists today and the proposal INTRODUCES the fan-out rather than being forced into it; each IsXxxMet has hand-fitted semantics. Failure mode: the generic evaluator ships, Status breaks, a special-case gets bolted back into the "generic" loop, hidden inside it and less honest than the switch it replaced. The hot table triples for a schema whose only proven consumers are Latency + one scalar. AlertFor deleted one-way, so band/range cases arriving later have no enum to hang per-type validation on, and AlertValue stays stringly-typed so direction and value-shape can silently disagree. Root cause: assuming severity comparison is universally reducible to a monotonic scalar + direction. Status is the counterexample already living in the switch. Fix: keep AlertFor as a discriminator OR replace with a Dimension string PLUS a ComparisonKind (Equality | Threshold), not Direction alone; generic threshold eval only for numeric dimensions; Status stays on its equality path; do NOT migrate CheckDataPoint now. Confidence: high.
First Principles high
The binding unit isn't a dimension name, it's a predicate: an alert config's job is to turn a raw result into "should I fire?" A dimension+direction+threshold is just one cheap encoding of a predicate (the monotonic-scalar case). Status, band, and equality are predicates too; they don't fit that encoding. Sound shape: keep the DimensionSpec registry (dedup is a pure win), but bind AlertConfig to a small discriminated union of rule-kinds where Threshold(direction) is one variant among several (StatusIs, Band, Equals). Don't delete the enum outright; don't bundle the reversible refactor with the irreversible migration — ship dedup + string dimension + generic loop behind the existing enum first (map enum->name internally), prove it, then delete and migrate. Verdict: directionally right, two structural flaws — ThresholdDirection too narrow to be the sole comparator, and reversible bundled with irreversible. Confidence: high.
Expansionist high
Two-lane split: Status stays a first-class predicate (StatusPredicate); dimensions are the numeric lane only. This resolves the sharpest fault (Status isn't numeric, has no ThresholdDirection) at near-zero cost, because it REMOVES work — Status never enters the dimension abstraction, so there's no forced impedance match. The user's option B ("keep Status out") is the two-lane split in embryo, but framed as a carve-out/exception; promoting it to the organizing principle is why it costs less — no special-casing later. Option A adopts a model that will need B bolted on anyway. Alternatives that ride on top cheaply: comparator-as-data (each DimensionSpec carries >=, <=, ==, in-band); typed AlertValue (StatusMatch | NumericThreshold) instead of a parsed string. Keep the reusable DimensionSpec (dedup, kill the enum) for the numeric lane; model Status separately. Confidence: high.
Outsider medium
Reading this as a wildlife field biologist running standardized surveys, where you must decide what counts as a measurement, its units, and what direction of change matters BEFORE collecting millions of irreversible observations. You're changing a data model and a hot storage schema at once, and the schema change is the irreversible one — different risk classes that should never ride the same decision. "Direction" is doing all the work, but you already named cases it can't hold: Status is a state, not a magnitude; "within a range" and "equals X" are real. In survey design that's the difference between continuous, categorical, and interval variables — you never force them onto one comparison operator. Storing the threshold as a parsed string is a units violation: a number without its unit and type isn't yet data. Prescription: adopt named measurements + generic eval now (reversible, cheap), but model comparison as a small typed set from day one (ordinal / range / categorical / equality). Pilot the storage migration on one metric with a dual-write window; don't decommission the old column until the new format survives a full cycle; don't delete the enum until the string names have earned reliability. Confidence: medium.

RICE: B~29, "missed E" (SDK + generic evaluator, AlertFor retired-not-deleted, string-compatible) ~19, "missed F" (two-PR split) ~15, D~10, A~2.0, C~1.8. Confidence: medium (high on B/E being correct).
