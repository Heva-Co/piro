# RFC 0001 — Third-party alert ingestion (Alertmanager, GCP Cloud Monitoring)

Status: proposal
Author: Arael Espinosa (assisted draft)
Date: 2026-07-13 (revised 2026-07-14)

## 1. Problem

Piro checks services operating at L5-L7 (HTTP, TCP, DNS, Ping, SSL, GRPC) from the outside in. This has two structural limits:

1. **Network reachability.** A check needs the target to be reachable from Piro's worker (embedded or remote). Resources living in a fully private network (k8s pods without an Ingress, internal services without egress) can't be checked without exposing them — which is often not acceptable for security reasons.
2. **Blindness to internal metrics.** Even if a service is reachable and returns `200 OK`, that status code says nothing about internal conditions: disk full, memory pressure, growing job queue, elevated error rate on a specific endpoint, etc. An HTTP check can't see that without the service exposing it ad-hoc — which is exactly what Prometheus (and, natively for GCP resources, Cloud Monitoring) already solves with exporters/metrics and alerting rules/policies.

Two concrete platforms motivate this RFC: a Prometheus + Alertmanager stack (e.g. in a k8s cluster), and Google Cloud Monitoring/Alerting (e.g. Cloud Run memory pressure, error rate, or any other GCP-native metric). Both already know how to evaluate "is this metric bad enough to alert" — Piro shouldn't reimplement that.

## 2. Non-goal (for now)

This proposal does **not** cover Piro actively pulling metrics (PromQL against Prometheus/Thanos/Mimir, or the Cloud Monitoring `timeSeries` API against GCP) to evaluate thresholds itself. That direction was evaluated and rejected for this phase for two reasons:

- It doesn't solve the motivating case (private network with no connectivity toward Piro) for the Prometheus scenario.
- For both Prometheus and GCP Cloud Monitoring, the target platform **already has its own alerting engine** (Alertmanager; GCP Alerting policies) that already decides severity/thresholds. Piro re-evaluating "is 90% memory bad?" itself would duplicate that work and create two places where the threshold can drift out of sync.

It can be evaluated as a separate RFC if a use case appears where Piro does have line of sight to the metrics backend and wants an ad-hoc "metric > threshold" check without an alerting rule already configured on the source platform.

## 3. Design principle

**Don't reinvent the source platform's alerting pipeline.** Alertmanager already handles routing, grouping, deduplication, silencing, and inhibition of alerts generated by PromQL rules. GCP Alerting already evaluates its own policies/conditions. Piro should not reimplement rule evaluation or grouping — it should limit itself to:

- Receiving the notifications the source platform already decided to send (Alertmanager's [standard webhook receiver](https://prometheus.io/docs/alerting/latest/configuration/#webhook_config); GCP Cloud Monitoring's [webhook notification channel](https://docs.cloud.google.com/monitoring/support/notification-options#webhooks)).
- Translating them into `Alert` — the same entity that Piro's internal checks generate today when crossing `FailureThreshold` — **not** directly into an `Incident`.

Promoting an `Alert` to an `Incident` (what makes it visible on the public status page and triggers `INotificationDispatcher`) remains, as it does today for internal alerts, an explicit human decision from the admin panel (`POST /api/v1/alerts/{id}/incident`). Neither webhook source may skip that step: it creates the signal, not the incident. This keeps a single "signal → human triage → public incident" pipeline without a parallel route that bypasses it, and is consistent with the fact that there's no automatic `Incident` creation from a failed check today either.

This makes the solution source-agnostic on the receiving end: anything Alertmanager or GCP Alerting can send reaches Piro through the same downstream pipeline (`Check` → `AlertConfig` → `AlertLifecycleService`), even though each source needs its own endpoint and payload parser (see §4.4/§4.7) — Piro doesn't need to understand PromQL or GCP's metric catalog at all, only each platform's already-decided notification contract.

## 4. Design

### 4.0 Relationship to Heartbeat ([issue #1](https://github.com/Heva-Co/piro/issues/1))

`CheckType.Heartbeat` is already planned (v0.4) and still unimplemented — it's the same underlying pattern: a **passive** check that Piro doesn't execute, but instead receives a push signal from outside. The difference is the semantics of that signal:

| | Heartbeat | `Webhook` (this RFC) |
|---|---|---|
| Signal | "still alive" (no status payload) | explicit `firing`/`resolved` event with severity |
| Transition to DOWN | on **absence** of a signal within an interval + grace period | on **presence** of a `firing` event |
| Requires a timeout evaluator | yes, essential (nothing tells you when something stops happening) | not in the base case — but worth adding anyway (see Risks, §8) to detect if the source platform itself goes silent |

Recommendation: when Heartbeat is implemented, extract the "last-signal-received evaluator" mechanism (the hosted service the issue mentions) as a reusable piece, and have `Webhook` consume it too for its "source went silent" case, instead of building a second timeout evaluator in parallel. Both `CheckType`s are specific cases of the same "passive check with an ingestion endpoint" concept — worth evaluating during implementation whether a shared base (e.g. a shared `LastSignalAt` field/table) makes sense instead of two isolated features.

### 4.1 Flow

```
Prometheus (alerting rules)                    GCP Cloud Monitoring (alerting policies)
   → Alertmanager (routing, grouping,             → evaluates policy/condition
      silencing)                                     ↓
      ↓                                           POST webhook notification channel
   POST webhook_config                              ↓
      ↓                                              ↓
      └──────────────────┬───────────────────────────┘
                          ↓
                 Piro.Api  [one endpoint per source, §4.4/§4.7]
                          ↓
                validates auth (per-source mechanism)
                          ↓
              maps alert → "external" Check (by correlation field) → associated AlertConfig
                          ↓
              creates/resolves Alert via AlertLifecycleService (dedup by fingerprint/incident id)
                          ↓
              (no human involvement — the automatic path ends here)
                          ↓
              admin reviews the Alert and decides whether to promote it to an Incident (manual, as today)
                          ↓
              that promotion triggers the existing INotificationDispatcher
```

Piro never initiates a connection toward the source platform or the private network behind it — the source is the one pushing. This solves the "private worker with no visibility" case for Prometheus/Alertmanager because no Piro→cluster connectivity is required, only cluster→Piro (which usually already exists for outbound egress). For GCP Cloud Monitoring the network argument is moot (it's already a public Google-managed service), but the "don't reinvent the alerting engine" argument (§3) still fully applies.

**Important**: there is no automatic notification just from receiving a webhook. `INotificationDispatcher` fires exactly as it does today, only when someone promotes the `Alert` to an `Incident` from the admin panel (or, in a later phase, if the team configures auto-promotion — see §6 Phase 3). This avoids the source platform's flapping spamming notification channels without human judgment in between — which would itself be a poor reinvention of the work Alertmanager/GCP Alerting already do with their own repeat/renotify intervals, but without its real severity context.

### 4.2 The mandatory `Check` problem

`Alert.CheckId` is a non-nullable `int` with an `OnDelete(Cascade)` FK — there's currently no path to create an `Alert` without a real `Check` behind it. `AlertLifecycleService.RecordOccurrenceAsync` assumes an `AlertConfig` evaluation already ran against a concrete `Check`. An alert coming from Alertmanager or GCP Cloud Monitoring has no such `Check` — the check itself was performed by the source platform, not Piro.

Rather than relaxing that FK (see §7.1 for the full comparison against this alternative, evaluated and rejected), the proposal is to introduce a **`CheckType.Webhook`**: a Check that never runs via `ICheckExecutor` (no `RoutingCheckJobDispatcher` ever triggers it — it has no `Cron`, or an inactive one) whose only role is to be the structural anchor for `Alert`s arriving via webhook. One is created per `Service` × external-source combination that should be monitored this way (typically one per Service, unless the team wants to split by alert type, policy, or resource).

This is consistent with the existing model: `Check.CurrentStatus` is still assigned directly (here, by the webhook instead of an executor), and it participates in `Service.CurrentStatus` aggregation via `ServiceStatusService` the same way, without touching that logic.

### 4.3 New integration types

Extend `IntegrationType` with **two** values, one per source (each source has a different auth mechanism and payload shape, so they don't share one type):

- **`AlertmanagerWebhook`** — `ConfigJson` stores an HMAC secret (or bearer token) generated by Piro when the integration is created, shown once in the admin panel — same UX pattern as `ApiKeyService` (only the hash is persisted).
- **`GcpMonitoringWebhook`** — `ConfigJson` stores a token used as a query-string parameter (GCP's webhook notification channel supports only Basic Auth or a `?auth_token=...` query parameter — no custom headers, no HMAC signature; see §4.7).

Both reuse the existing `Integration` entity instead of creating new tables — consistent with the fact that `Check.IntegrationId` today is already "a config blob associated with a monitoring entity", just in the reverse direction (outbound today, inbound for this case).

### 4.4 Alertmanager endpoint

`POST /api/v1/webhooks/alertmanager/{integrationId}`

- **Authentication**: `Authorization: Bearer <secret>` header validated against the hash stored on the `Integration`, mirroring `ApiKeyService.ValidateAsync` (SHA-256 hash comparison, never the plaintext secret). A more robust alternative if plaintext secrets traveling in a header is a concern: an HMAC-SHA256 signature over the body with a shared secret (standard webhook pattern, e.g. Stripe/GitHub) — recommended if multi-tenant rotatable secrets without downtime are supported in the future; for v1, a simple bearer token is sufficient and consistent with what already exists in the codebase.
- **Payload**: the standard JSON format Alertmanager sends to a `webhook_config` (`version`, `status`, `alerts[]`, each alert with `labels`, `annotations`, `startsAt`, `endsAt`, `generatorURL`). No transformation is required on the Prometheus/Alertmanager side — it's the native payload, no plugins.
- **Correlation field**: the alert must include a `piro_check_slug` label (configured on the Prometheus rule / on the Alertmanager `route`) that Piro resolves against a stable identifier of the `Check`. This avoids guessing the mapping from the alert name — Alertmanager lets an operator inject arbitrary custom labels on a rule, so this is a label Piro invents and the operator sets deliberately.
- **Idempotency**: Alertmanager periodically resends notifications while an alert is active (`repeat_interval`) and on resolution (`resolved`). `AlertLifecycleService.RecordOccurrenceAsync` is already designed for this — it dedupes by `MessageFingerprint` and increments `OccurrenceCount` instead of creating a new `Alert`. Alertmanager's fingerprint (deterministic per label set) feeds that `MessageFingerprint`.

### 4.5 Translation into the domain model (both sources)

- A firing alert (`status: "firing"` for Alertmanager; `incident.state: "open"` for GCP) → resolve the `Check` (`CheckType.Webhook`) via the source's correlation field, resolve its `AlertConfig` (one per Check, as already enforced today), and call `AlertLifecycleService.RecordOccurrenceAsync` the same way `AlertEvaluationService` would after an internal execution — it creates the `Alert` if there's no active one with the same fingerprint, or increments `OccurrenceCount` if there already is. `Check.CurrentStatus` is updated to the mapped severity (`critical→DOWN`, `warning→DEGRADED`, default `DEGRADED` if absent), participating in `ServiceStatusService`'s normal aggregation.
- A resolved alert (`status: "resolved"`; `incident.state: "closed"`) → resolve the same `Check`/fingerprint and call `AlertLifecycleService.ResolveActiveAlertAsync` — the same call site the internal automatic resolution uses, nothing reimplemented.
- **The `Alert` is left with `IncidentId = null`.** No `Incident` is created or touched in this flow. It remains human work: someone reviews it in the admin panel (`AlertsOverviewController`) and decides `POST /api/v1/alerts/{id}/incident` if it warrants public communication — exactly the same path that already exists for alerts generated by internal checks.
- `INotificationDispatcher` does not fire just because the webhook arrived — only when (and if) that `Alert` gets promoted to an `Incident`, same as today.

### 4.6 What does NOT change

- `ICheckExecutor` — untouched; the `Webhook`-type `Check` simply has no executor registered (same behavior that already exists today for a `CheckType` with no executor: `LocalCheckJobDispatcher` already handles that case with `DataType = MONITOR_OUTAGE`, though here it never fires because it has no active `Cron`).
- `AlertConfig`, `AlertLifecycleService`, `AlertEvaluationService` — reused without modifying their logic; each webhook source is an additional source of "occurrences", not a parallel pipeline.
- `IncidentAppService`, `INotificationDispatcher` — untouched; promotion to an incident stays 100% manual, as today.
- `CheckDataPoint` schema — not applicable, this flow doesn't generate check time series.

### 4.7 GCP Cloud Monitoring endpoint

`POST /api/v1/webhooks/gcp/{integrationId}` — a **separate** endpoint from Alertmanager's, not a shared one with format auto-detection. The two sources differ enough (auth mechanism, payload shape, correlation field) that mixing both parsers behind one handler would need ugly conditional logic; a dedicated endpoint per source keeps each parser/auth contract simple and independently testable.

- **Authentication**: GCP's webhook notification channel [only supports two mechanisms](https://docs.cloud.google.com/monitoring/support/notification-options#webhooks) — Basic Auth (server challenges with `401`/`WWW-Authenticate` per RFC 2617) or a token passed as a query-string parameter (`?auth_token=<secret>`). There is no HMAC/signature option and no custom-header support. **Decision: token in the query string** — at the cost of the secret being present in the request URL, this requires explicit care: the query parameter must be scrubbed from any HTTP access/request logging middleware (same class of risk the RFC already flags for Alertmanager's bearer token in §8, but here it's the *only* available option rather than one of two).
  - **Console UX note**: the Cloud Console's "Add new webhook" dialog only exposes a single "Endpoint URL" field — there is no separate credentials field for the token, unlike Basic Auth (which does get dedicated username/password fields in the same dialog). The token is configured by embedding it directly in that URL field (e.g. `https://<piro-host>/api/v1/webhooks/gcp/{integrationId}?auth_token=<secret>`), not through a distinct auth UI. The admin UX in Phase 3 (§6) should pre-build and display this full URL (webhook path + query token already appended) for the operator to copy-paste directly into that one field, rather than expecting them to piece the token into the URL themselves.
- **Payload**: GCP's [version 1.2 webhook schema](https://docs.cloud.google.com/monitoring/support/notification-options#webhooks) — a single top-level `incident` object per request (not an array like Alertmanager), containing (among other fields) `incident_id`, `state` (`"open"`/`"closed"`), `started_at`/`ended_at`, `resource` (`type` + `labels`, shape depends on `resource.type`), `resource_display_name`, `metric` (`type`, `labels`), `policy_name`, `condition_name`, `severity`, `observed_value`/`threshold_value`, `documentation`, `summary`.
- **Correlation field**: unlike Alertmanager, GCP has no mechanism for an operator-defined free-form label on a condition. The obvious first guess — `resource.labels.service_name` — does **not** generalize: `resource.labels`' shape depends entirely on `resource.type` (a `cloud_run_revision` has `service_name`; a `gce_instance` has `instance_id`/`zone` instead, no `service_name` at all). A mapping keyed to one `resource.labels` field would silently stop working the moment a policy covers a different GCP resource type. Two fields are stable across every resource type instead: `policy_name` (chosen by the operator when creating the alert policy in GCP, so it can deliberately be set to match a Check's identifier — the same intent as `piro_check_slug`) and `resource_display_name` (present regardless of `resource.type`, but maps one `Check.Webhook` per concrete resource rather than per policy). **Decision: correlate by `policy_name`** — it groups multiple resources under one policy into a single Piro `Check` when that's the intended granularity, and mirrors the deliberate-operator-chosen-identifier pattern Alertmanager already uses.
- **Idempotency**: GCP doesn't document a `MessageFingerprint`-equivalent concept explicitly, but `incident.incident_id` is a stable, GCP-generated identifier per incident (the same incident's `open`→`closed` lifecycle and any `renotify: true` repeats share the same `incident_id`) — **use `incident_id` directly as `MessageFingerprint`**, the same dedup mechanism `AlertLifecycleService` already provides for Alertmanager.
- **Severity mapping**: GCP's `incident.severity` field (e.g. `"critical"`, `"warning"`, `"no severity"` as seen in test payloads) maps the same way Alertmanager's `labels.severity` does — `critical→DOWN`, `warning→DEGRADED`, default `DEGRADED` if absent/unrecognized (e.g. `"no severity"`).

## 5. Data / schema scope

Schema changes required (to be validated during implementation):

- `IntegrationType`: add `AlertmanagerWebhook` and `GcpMonitoringWebhook`.
- `CheckType`: add `Webhook` (a Check that never executes, only anchors incoming `Alert`s from either source).
- `Check`: no column changes — a `Webhook` Check uses `TypeDataJson` to store the correlation value expected from its source (`piro_check_slug` value for Alertmanager, `policy_name` value for GCP) and can leave `Cron` at a value that never fires, or add a simple flag if explicitness is preferred (`IsPassive`, to be evaluated during implementation) instead of inferring it from `CheckType`.
- No changes to `Alert`, `AlertConfig`, `Incident`, `Service`, `CheckDataPoint` — reused as-is. (`Alert.CheckId` explicitly stays non-nullable — see §7.1.)

## 6. Phased plan

1. **Phase 1 — Alertmanager basic ingestion**: `CheckType.Webhook`, `/webhooks/alertmanager/{integrationId}` endpoint + bearer authentication + `Alert` creation/resolution via `AlertLifecycleService` by fingerprint, mandatory `piro_check_slug` label mapping. No automatic promotion to `Incident`. No configuration UI yet — the `Integration` and `Check` are created via API/seed.
2. **Phase 2 — GCP Cloud Monitoring ingestion**: `/webhooks/gcp/{integrationId}` endpoint + query-token authentication + `incident_id`-based dedup + `policy_name` correlation, reusing the same `CheckType.Webhook`/`AlertConfig`/`AlertLifecycleService` pipeline from Phase 1.
3. **Phase 3 — Admin UX**: screen in `apps/admin` to create either integration type, generate and show the secret/token once, create the `Webhook`-type `Check`(s) with their `AlertConfig`, copy the webhook URL (with the correlation value pre-filled) and, for Alertmanager, a ready-to-paste config snippet. Incoming `Alert`s show up in the existing `AlertsOverviewController`/alerts view, with no special distinction beyond their `Check.Type`.
4. **Phase 4 — Optional auto-promotion**: allow configuring, per integration, a `severity → auto-promote to Incident` mapping (e.g. `critical` always creates+publishes an `Incident` without waiting on the admin) for teams that trust their upstream alerting rules and want the status page reacting without human intervention. This is an explicit opt-in flag, not the default behavior.
5. **Phase 5 (optional, out of scope for this RFC)**: evaluate whether it's worth adding a metrics-pull direction (PromQL against Prometheus, or the Cloud Monitoring `timeSeries` API against GCP) as an additional `CheckType`, for cases where Piro does have line of sight to the metrics backend and a "metric > threshold" check is wanted without an alerting rule already configured there.

## 7. Alternatives considered

- **Prometheus remote_write into a receiver in Piro**: rejected — it implies Piro understanding the remote_write protocol (protobuf + snappy) and storing time series, which reinvents what Prometheus/Thanos already do. Unnecessary if the goal is "act accordingly", not "graph metrics".
- **Piro as a Prometheus exporter** (`/metrics` on Piro.Api for an external Prometheus to scrape Piro): a different problem (observability of Piro itself, not of the services Piro monitors) and doesn't solve the motivating case. Could be a separate RFC if the goal is to feed the status page with the platform's own metrics.
- **Metrics pull per check** (PromQL, or the GCP Cloud Monitoring `timeSeries` API — see §2): rejected for this phase because it duplicates threshold-evaluation work the source platform's own alerting engine already does, and (for Prometheus specifically) doesn't solve the private-network motivating case.

### 7.1 `CheckType.Webhook` anchor vs. a nullable `Alert.CheckId`

Once a second webhook source (GCP Cloud Monitoring, in addition to Alertmanager) was on the table, the question of whether third-party-sourced alerts really need a `Check` behind them at all came up again. Two options were weighed directly against each other:

**Option A — keep the `CheckType.Webhook` anchor (this RFC's design, §4.2)**: every third-party alert still resolves to a real `Check` row (one per Service × external source, non-executing) before `AlertLifecycleService` touches it.

**Option B — make `Alert.CheckId` nullable**, letting an `Alert` exist with `CheckId = null` and (if adopted) a direct `Alert.ServiceId`, skipping the anchor `Check` entirely.

A full codebase audit (backend + both frontends) was run specifically to evaluate Option B before deciding. Findings:

| | Option A — `Check.Webhook` anchor | Option B — nullable `Alert.CheckId` |
|---|---|---|
| Schema/migration risk | None — reuses `Check`/`AlertConfig` as-is, no FK changes | New migration; `AlertEntityConfiguration`'s FK must move from `OnDelete(Cascade)` to `OnDelete(SetNull)` (mirroring `IncidentServiceConfiguration`'s existing `TriggeringCheckId` pattern) |
| Call sites requiring a code change | Zero — the rest of the system already assumes every `Alert` has a `Check` | At least 7, several high-severity: `EscalationCheckerService.cs` dereferences `alert.Check.Name`/`.Slug` unguarded in 6 places inside `ProcessAlertAsync` (runs every minute per active alert) — a check-less alert reaching an escalation-enabled service would throw `NullReferenceException` there (contained per-alert by an existing try/catch, but that alert would then silently never escalate again); `AlertAppService.cs`'s `IncidentTitleFactory.Build(alert.Check.Type)` has the same unguarded-access problem on the "promote to Incident" path |
| Silent-data-loss risk | None | `AlertRepository.cs`'s list/detail projections (`a.Check.Slug`, `a.Check.Name`) currently assume an inner join; without care, a check-less alert either fails to compile or silently disappears from the paged list / 404s on direct fetch — exactly the "invisible alert" failure mode this feature is meant to avoid |
| DTO/API contract impact | None | `AlertSummaryDto`/`AlertDetailDto`'s `CheckSlug`/`CheckName` are non-nullable `string` today — becoming nullable ripples into the OpenAPI-generated frontend types and requires null-guards in `apps/admin`'s `AlertsPage.tsx`/`AlertDetailPage.tsx` (the codebase already has one correct precedent to copy: `IncidentExtensions.cs`'s `a.Check?.Slug ?? a.CheckId.ToString()`, plus `IncidentService.TriggeringCheckId`, which is already nullable with `SetNull`) |
| `Service.CurrentStatus` participation | Automatic — the anchor `Check` already participates in `ServiceStatusService`'s aggregation like any other Check, no special-casing | Undefined by default — a check-less alert has no natural place in the existing Check-based aggregation; would need new logic to decide whether/how it affects `Service.CurrentStatus` |
| New ingestion capability still required either way | Yes | Yes — `AlertLifecycleService.RecordOccurrenceAsync` requires a concrete `Check` parameter regardless of the option chosen; neither option is "free", a new creation path has to be built for both |
| Setup friction | One `Integration` + one `Check.Webhook` + one `AlertConfig` per Service/external source to configure (mitigated by the Phase 3 admin UX, §6) | None beyond the `Integration` itself |

**Decision: kept Option A.** Given that a new ingestion path is unavoidable either way (second-to-last row), Option B's only real advantage is removing the one-time setup step of creating an anchor `Check` — which Phase 3's admin UX already collapses into a single guided flow — while its cost is touching the most sensitive piece of business logic in the system (`EscalationCheckerService`) plus several silent-data-loss traps. The setup friction is a one-time, UX-solvable cost; the nullable-FK option's risk is a standing maintenance burden across every future change to Alert-adjacent code, since every new `alert.Check.X` access anywhere in the codebase would need to remember the check-less case forever.

## 8. Risks

- **Reliability of the correlation field**: if a team doesn't configure `piro_check_slug` on their Prometheus rules, or doesn't set `policy_name` to match a Piro Check on a GCP alert policy, the alert can't be resolved to a `Check` — an internal log/metric for "webhook alerts without a resolvable correlation field" should be recorded to catch this during setup, and it should be clearly documented in the wiki for both sources.
- **Resend storms**: Alertmanager periodically resends active notifications (`repeat_interval`, default 4h but configurable down to minutes); GCP resends with `renotify: true` on an unspecified schedule. Without the dedup by fingerprint (`MessageFingerprint` for Alertmanager, `incident_id` for GCP) that `AlertLifecycleService` already provides, each resend would create a new `Alert` instead of incrementing the existing one's `OccurrenceCount`, inflating noise in the admin's alerts view.
- **A `Webhook` Check with no alerts never looks "green" the same way an active check does**: since it never executes, its `CurrentStatus` stays at the last received value indefinitely if the source platform stops sending resolution notifications (e.g. if Alertmanager or the GCP notification channel itself breaks). Unlike a normal check, there's no heartbeat of its own to detect that — worth considering, during implementation, a TTL or a "last signal received more than X ago" check so silence isn't trusted as a health signal.
- **Secret exposed in logs**: Alertmanager's bearer token must never be logged by the HTTP request logging middleware, same care already given to `ApiKey` headers. For GCP, this risk is unavoidable by design (the auth token is GCP's only supported mechanism besides Basic Auth, and it travels in the query string) — the query parameter must be explicitly scrubbed from any access/request logs, not just from application-level logging.
- **GCP's payload shape depends on `resource.type`**: any code that reads `resource.labels.*` fields directly (rather than the type-independent `policy_name`/`resource_display_name`/`incident_id`/`severity` fields used by this design) will break silently for resource types it wasn't tested against. Confirmed during design by comparing a real Cloud Run (`cloud_run_revision`) payload against a GCP-documented VM instance (`gce_instance`) example — the two have entirely different `resource.labels` shapes. The top-level `resource_id` field has the same problem and was considered and rejected as a correlation field for the same reason: the docs define it verbatim as "instance ID of the monitored resource, same value as `resource.labels.instance_id`" — i.e. it's defined in terms of a label (`instance_id`) that itself only exists for `gce_instance`-type resources. **Confirmed empirically** by triggering a real test alert policy (`instance_count` on a live Cloud Run service, `hevaplatform` project) against a webhook.site endpoint: for a `cloud_run_revision` resource, `resource_id` arrives as an **empty string** (`"resource_id": ""`), while `policy_name` arrives as the exact operator-chosen display name (e.g. `"POC-RFC0001-DELETE-ME"`, not the full `projects/.../alertPolicies/{id}` path) — validating both the rejection of `resource_id` and the choice of `policy_name` as the correlation field. The real payload's `resource.labels` for `cloud_run_revision` only included `project_id`/`service_name` (not `revision_name`/`location`/`configuration_name`, which appear in the separate `timeSeries` API but not in this webhook payload) — another reason to avoid keying correlation logic on `resource.labels` contents, since even the same resource type's label set can vary by API surface.
