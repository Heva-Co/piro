# RFC 0001 — Prometheus Alertmanager Integration (inbound webhook)

Status: proposal
Author: Arael Espinosa (assisted draft)
Date: 2026-07-13

## 1. Problem

Piro checks services operating at L5-L7 (HTTP, TCP, DNS, Ping, SSL, GRPC) from the outside in. This has two structural limits:

1. **Network reachability.** A check needs the target to be reachable from Piro's worker (embedded or remote). Resources living in a fully private network (k8s pods without an Ingress, internal services without egress) can't be checked without exposing them — which is often not acceptable for security reasons.
2. **Blindness to internal metrics.** Even if a service is reachable and returns `200 OK`, that status code says nothing about internal conditions: disk full, memory pressure, growing job queue, elevated error rate on a specific endpoint, etc. An HTTP check can't see that without the service exposing it ad-hoc — which is exactly what Prometheus already solves with exporters and alerting rules.

## 2. Non-goal (for now)

This proposal does **not** cover Piro actively pulling PromQL from a Prometheus/Thanos/Mimir instance. That direction was ruled out for this phase because it doesn't solve the motivating case (private network with no connectivity toward Piro) and adds a per-check configuration surface (endpoint, query, credentials) that isn't necessary if the source of truth for alerting is already Alertmanager. It can be evaluated as a separate RFC if a use case appears where Piro does have line of sight to the Prometheus instance.

## 3. Design principle

**Don't reinvent Prometheus's alerting pipeline.** Alertmanager already handles routing, grouping, deduplication, silencing, and inhibition of alerts generated by PromQL rules. Piro should not reimplement rule evaluation or grouping rules — it should limit itself to:

- Receiving the notifications Alertmanager already decided to send (via its [standard webhook receiver](https://prometheus.io/docs/alerting/latest/configuration/#webhook_config)).
- Translating them into `Alert` — the same entity that Piro's internal checks generate today when crossing `FailureThreshold` — **not** directly into an `Incident`.

Promoting an `Alert` to an `Incident` (what makes it visible on the public status page and triggers `INotificationDispatcher`) remains, as it does today for internal alerts, an explicit human decision from the admin panel (`POST /api/v1/alerts/{id}/incident`). The Alertmanager webhook must not skip that step: it creates the signal, not the incident. This keeps a single "signal → human triage → public incident" pipeline without a parallel route that bypasses it, and is consistent with the fact that there's no automatic `Incident` creation from a failed check today either.

This makes the solution source-agnostic: anything Alertmanager can receive (Prometheus, Node Exporter, cAdvisor, kube-state-metrics, GCP/AWS exporters, custom rules) reaches Piro through the same path, without Piro needing to understand Prometheus at all — only Alertmanager's JSON contract.

## 4. Design

### 4.0 Relationship to Heartbeat ([issue #1](https://github.com/Heva-Co/piro/issues/1))

`CheckType.Heartbeat` is already planned (v0.4) and still unimplemented — it's the same underlying pattern: a **passive** check that Piro doesn't execute, but instead receives a push signal from outside. The difference is the semantics of that signal:

| | Heartbeat | `Webhook` (this RFC) |
|---|---|---|
| Signal | "still alive" (no status payload) | explicit `firing`/`resolved` event with severity |
| Transition to DOWN | on **absence** of a signal within an interval + grace period | on **presence** of a `firing` event |
| Requires a timeout evaluator | yes, essential (nothing tells you when something stops happening) | not in the base case — but worth adding anyway (see Risks, §8) to detect if Alertmanager itself goes silent |

Recommendation: when Heartbeat is implemented, extract the "last-signal-received evaluator" mechanism (the hosted service the issue mentions) as a reusable piece, and have `Webhook` consume it too for its "Alertmanager went silent" case, instead of building a second timeout evaluator in parallel. Both `CheckType`s are specific cases of the same "passive check with an ingestion endpoint" concept — worth evaluating during implementation whether a shared base (e.g. a shared `LastSignalAt` field/table) makes sense instead of two isolated features.

### 4.1 Flow

```
Prometheus (alerting rules)
   → Alertmanager (routing, grouping, silencing)
      → POST webhook  →  Piro.Api  [new endpoint]
                              ↓
                    validates signature/secret via Integration
                              ↓
                    maps alert → "external" Check (by label) → associated AlertConfig
                              ↓
                    creates/resolves Alert via AlertLifecycleService (fingerprint = dedup)
                              ↓
                    (no human involvement — the automatic path ends here)
                              ↓
                    admin reviews the Alert and decides whether to promote it to an Incident (manual, as today)
                              ↓
                    that promotion triggers the existing INotificationDispatcher
```

Piro never initiates a connection toward the cluster's network — Alertmanager is the one pushing. This solves the "private worker with no visibility" case because no Piro→cluster connectivity is required, only cluster→Piro (which usually already exists for outbound egress).

**Important**: there is no automatic notification just from receiving the webhook. `INotificationDispatcher` fires exactly as it does today, only when someone promotes the `Alert` to an `Incident` from the admin panel (or, in a later phase, if the team configures auto-promotion — see §6 Phase 3). This avoids Alertmanager flapping spamming notification channels without human judgment in between — which would itself be a poor reinvention of the work Alertmanager already does with `group_wait`/`repeat_interval`, but without its real severity context.

### 4.2 The mandatory `Check` problem

`Alert.CheckId` is a non-nullable `int` with an `OnDelete(Cascade)` FK — there's currently no path to create an `Alert` without a real `Check` behind it. `AlertLifecycleService.RecordOccurrenceAsync` assumes an `AlertConfig` evaluation already ran against a concrete `Check`. An alert coming from Alertmanager has no such `Check` — the check itself was performed by Prometheus, not Piro.

Rather than relaxing that FK (which would break the `Alert → Check → Service` invariant the rest of the system relies on, including resolving `TriggeringCheckId` on incidents), the proposal is to introduce a **`CheckType.Webhook`**: a Check that never runs via `ICheckExecutor` (no `RoutingCheckJobDispatcher` ever triggers it — it has no `Cron`, or an inactive one) whose only role is to be the structural anchor for `Alert`s arriving via webhook. One is created per `Service` × external-source combination that should be monitored via Alertmanager (typically one per Service, unless the team wants to split by alert type).

This is consistent with the existing model: `Check.CurrentStatus` is still assigned directly (here, by the webhook instead of an executor), and it participates in `Service.CurrentStatus` aggregation via `ServiceStatusService` the same way, without touching that logic.

### 4.3 New integration type

Extend `IntegrationType` with `AlertmanagerWebhook`. An `Integration` of this type stores in `ConfigJson`:

- An HMAC secret (or bearer token) generated by Piro when the integration is created, shown once in the admin panel — same UX pattern as `ApiKeyService` (only the hash is persisted).
- A mapping from Alertmanager labels to the corresponding `Webhook`-type `Check`. Minimum viable approach: the alert must include a `piro_check_slug` label (configured on the Prometheus rule / on the Alertmanager `route`) that Piro resolves against a stable identifier of the Check. This avoids guessing the mapping from the alert name.

This reuses the existing `Integration` entity instead of creating a new table — consistent with the fact that `Check.IntegrationId` today is already "a config blob associated with a monitoring entity", just in the reverse direction (outbound today, inbound for this case).

### 4.4 New endpoint

`POST /api/v1/webhooks/alertmanager/{integrationId}`

- **Authentication**: `Authorization: Bearer <secret>` header validated against the hash stored on the `Integration`, mirroring `ApiKeyService.ValidateAsync` (SHA-256 hash comparison, never the plaintext secret). A more robust alternative if plaintext secrets traveling in a header is a concern: an HMAC-SHA256 signature over the body with a shared secret (standard webhook pattern, e.g. Stripe/GitHub) — recommended if multi-tenant rotatable secrets without downtime are supported in the future; for v1, a simple bearer token is sufficient and consistent with what already exists in the codebase.
- **Payload**: the standard JSON format Alertmanager sends to a `webhook_config` (`version`, `status`, `alerts[]`, each alert with `labels`, `annotations`, `startsAt`, `endsAt`, `generatorURL`). No transformation is required on the Prometheus/Alertmanager side — it's the native payload, no plugins.
- **Idempotency**: Alertmanager periodically resends notifications while an alert is active (`repeat_interval`) and on resolution (`resolved`). `AlertLifecycleService.RecordOccurrenceAsync` is already designed for this — it dedupes by `MessageFingerprint` and increments `OccurrenceCount` instead of creating a new `Alert`. Alertmanager's fingerprint (deterministic per label set) feeds that `MessageFingerprint`.

### 4.5 Translation into the domain model

- A `firing` alert → resolve the `Check` (`CheckType.Webhook`) via the `piro_check_slug` label, resolve its `AlertConfig` (one per Check, as already enforced today), and call `AlertLifecycleService.RecordOccurrenceAsync` the same way `AlertEvaluationService` would after an internal execution — it creates the `Alert` if there's no active one with the same fingerprint, or increments `OccurrenceCount` if there already is. `Check.CurrentStatus` is updated to the mapped severity (`critical→DOWN`, `warning→DEGRADED`, default `DEGRADED` if absent), participating in `ServiceStatusService`'s normal aggregation.
- A `resolved` alert → resolve the same `Check`/fingerprint and call `AlertLifecycleService.ResolveActiveAlertAsync` — the same call site the internal automatic resolution uses, nothing reimplemented.
- **The `Alert` is left with `IncidentId = null`.** No `Incident` is created or touched in this flow. It remains human work: someone reviews it in the admin panel (`AlertsOverviewController`) and decides `POST /api/v1/alerts/{id}/incident` if it warrants public communication — exactly the same path that already exists for alerts generated by internal checks.
- `INotificationDispatcher` does not fire just because the webhook arrived — only when (and if) that `Alert` gets promoted to an `Incident`, same as today.

### 4.6 What does NOT change

- `ICheckExecutor` — untouched; the `Webhook`-type `Check` simply has no executor registered (same behavior that already exists today for a `CheckType` with no executor: `LocalCheckJobDispatcher` already handles that case with `DataType = MONITOR_OUTAGE`, though here it never fires because it has no active `Cron`).
- `AlertConfig`, `AlertLifecycleService`, `AlertEvaluationService` — reused without modifying their logic; the webhook is an additional source of "occurrences", not a parallel pipeline.
- `IncidentAppService`, `INotificationDispatcher` — untouched; promotion to an incident stays 100% manual, as today.
- `CheckDataPoint` schema — not applicable, this flow doesn't generate check time series.

## 5. Data / schema scope

Schema changes required (to be validated during implementation):

- `IntegrationType`: add `AlertmanagerWebhook`.
- `CheckType`: add `Webhook` (a Check that never executes, only anchors incoming `Alert`s).
- `Check`: no column changes — a `Webhook` Check uses `TypeDataJson` to store the correlation label (the expected `piro_check_slug`) and can leave `Cron` at a value that never fires, or add a simple flag if explicitness is preferred (`IsPassive`, to be evaluated during implementation) instead of inferring it from `CheckType`.
- No changes to `Alert`, `AlertConfig`, `Incident`, `Service`, `CheckDataPoint` — reused as-is.

## 6. Phased plan

1. **Phase 1 — Basic ingestion**: `CheckType.Webhook`, endpoint + bearer authentication + `Alert` creation/resolution via `AlertLifecycleService` by fingerprint, mandatory `piro_check_slug` label mapping. No automatic promotion to `Incident`. No configuration UI yet — the `Integration` and `Check` are created via API/seed.
2. **Phase 2 — Admin UX**: screen in `apps/admin` to create the integration, generate and show the secret once, create the `Webhook`-type `Check`(s) with their `AlertConfig`, copy the webhook URL and a ready-to-paste Alertmanager config snippet. Incoming `Alert`s show up in the existing `AlertsOverviewController`/alerts view, with no special distinction beyond their `Check.Type`.
3. **Phase 3 — Optional auto-promotion**: allow configuring, per integration, a `severity → auto-promote to Incident` mapping (e.g. `critical` always creates+publishes an `Incident` without waiting on the admin) for teams that trust their Prometheus rules and want the status page reacting without human intervention. This is an explicit opt-in flag, not the default behavior.
4. **Phase 4 (optional, out of scope for this RFC)**: evaluate whether it's worth adding the PromQL pull direction as an additional `CheckType`, for cases where Piro does have line of sight to Prometheus and a "metric > threshold" check is wanted without going through an Alertmanager rule.

## 7. Alternatives considered

- **Prometheus remote_write into a receiver in Piro**: rejected — it implies Piro understanding the remote_write protocol (protobuf + snappy) and storing time series, which reinvents what Prometheus/Thanos already do. Unnecessary if the goal is "act accordingly", not "graph metrics".
- **Piro as a Prometheus exporter** (`/metrics` on Piro.Api for an external Prometheus to scrape Piro): a different problem (observability of Piro itself, not of the services Piro monitors) and doesn't solve the motivating case. Could be a separate RFC if the goal is to feed the status page with the platform's own metrics.
- **PromQL pull per check** (see §2): rejected for this phase due to network topology and because it duplicates work Alertmanager already does.

## 8. Risks

- **Reliability of the label mapping**: if a team doesn't configure `piro_check_slug` on their rules, the alert can't be resolved to a `Check` — an internal log/metric for "Alertmanager alerts without check_slug" should be recorded to catch this during setup, and it should be clearly documented in the wiki.
- **Resend storms**: Alertmanager periodically resends active notifications (`repeat_interval`, default 4h but configurable down to minutes) — without the dedup by `MessageFingerprint` that `AlertLifecycleService` already provides, each resend would create a new `Alert` instead of incrementing the existing one's `OccurrenceCount`, inflating noise in the admin's alerts view.
- **A `Webhook` Check with no alerts never looks "green" the same way an active check does**: since it never executes, its `CurrentStatus` stays at the last received value indefinitely if Alertmanager stops sending `resolved` (e.g. if Alertmanager itself goes down). Unlike a normal check, there's no heartbeat of its own to detect that — worth considering, during implementation, a TTL or a "last signal received more than X ago" check so silence isn't trusted as a health signal.
- **Secret exposed in logs**: if a bearer token is used, make sure it's never logged by the HTTP request logging middleware — apply the same care already given to `ApiKey` headers.
